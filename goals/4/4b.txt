PART B — merge watchlist promotion into the resolved-track-record gate
====================================================
Today, sources/accounts.py's discover_winners()/_is_winner() correctly gates
on resolved positions only (min_resolved_count, min_win_rate, coin-flip/sports
exclusion). Separately, config.json's "accounts.watchlist" (20 hardcoded
addresses with a manually-entered, never-recomputed "monthly_pnl" figure)
feeds analysis/smart_money_scan.py -> core/scanner.py's tag_watchlist_overlap,
which gives watchlist_signal the #1 sort priority ahead of flag/net_edge, and
main.py promotes watchlist-tagged markets into flagged_markets even when they
didn't otherwise flag. The watchlist path never calls _is_winner and has no
resolved_count or win_rate check.

Fix: before a watchlist trader's open positions are allowed to set
watchlist_signal = True or influence sort/promotion, that trader's wallet
must independently clear sources/accounts._is_winner() against its OWN
resolved positions (same thresholds: min_resolved_count, min_win_rate,
min_cash_pnl, min_pct_pnl, same coin-flip/sports exclusion). Concretely:

  1. In analysis/smart_money_scan.py's fetch_watchlist_positions (or a new
     thin wrapper), for each watchlist entry call accounts.fetch_user_positions
     + accounts._score_wallet on that address, then accounts._is_winner(stats,
     config). Cache this verification result alongside the existing position
     cache (same TTL is fine, or its own — your call, just don't refetch on
     every call).
  2. If a watchlist trader fails _is_winner, their open positions must NOT set
     watchlist_signal / watchlist_direction / watchlist_position_val on any
     market in core/scanner.py, and must NOT be eligible for the
     flagged_markets promotion in main.py. Log clearly which traders were
     excluded and why, e.g.:
       "[watchlist] EXCLUDED <name> (<address>) — resolved_count=3 (<10
        required), win_rate=N/A — config-listed monthly_pnl is unverified
        and ignored"
  3. Traders who DO pass _is_winner keep full current behavior (sort
     priority, promotion into flagged_markets).
  4. The static "monthly_pnl" field in config.json stays for human reference
     only (it is what got these addresses onto the list in the first place)
     but must never again be the sole basis for a signal-affecting decision.
     Add a one-line comment in config.json above "watchlist" stating this.
  5. Do not delete or shrink the watchlist list itself — just gate what it's
     allowed to do downstream.

This directly intersects backlog.json's existing locked items
"per-wallet-track-record" and "skill-vs-luck-weighting" and the blocked

PART C — fee-aware edge at signal-generation time
====================================================
Today, core/scanner.py computes net_edge = raw_edge - half_spread with no fee
term, and core/report.py's _ev_per_contract uses the same fee-free price.
Fees are only ever applied in core/logger.py's resolve_outcomes(), and only
for source == 'real_fill' rows — paper signals report fee-free edge forever.

Add a pure function (core/scanner.py or a new core/fees.py, your call) that
computes Kalshi's per-contract fee given price and contract count using
Kalshi's published fee schedule (round up, standard formula —
fee ~= ceil(0.07 * price * (1 - price) * contracts * 100) / 100 in dollars;
verify the exact current multiplier/rounding against Kalshi's fee docs rather
than assuming this is exact, and cite where you got it in a code comment).

Use this to compute a fee-adjusted net_edge_after_fee and
ev_after_fee_per_contract at the SAME point net_edge is currently computed
in scanner.py, store both the existing fee-free fields (do not remove them —
they may be referenced elsewhere) and the new after-fee fields on the signal
dict, and persist the after-fee fields as new columns in the signals table
(migrate leviathan.db: back up to leviathan.db.bak_4b first, additive
columns only, no data loss, existing rows can have NULL for the new columns).

Wire the after-fee EV into the EV floor filter from PART A — the floor in
_betting_queue must check ev_after_fee, not the fee-free ev, since the
fee-free number is what's currently overstating real opportunity.

Produce the printed before/after comparison required by the proof criteria:
pick one real signal currently in leviathan.db, show its fee-free net_edge
and EV, its fee-adjusted net_edge and EV, and the dollar/percentage delta.
