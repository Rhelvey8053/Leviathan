PART B — backlog.py (CLI, importable engine)

Shared functions (importable for the weekly checker later and for tests):
  - parse_trigger(s) -> dict. "manual" returns {"all": []}. Otherwise 
    split on comma, parse each "metric<op>value" via regex 
    ^([a-z_]+)(>=|<=|==|>|<)(\d+(?:\.\d+)?)$, value cast to number.
  - determine_status(trigger, depends_on) -> applies precedence above.
  - validate_item(item, existing_items, glossary) -> returns list of 
    error strings (empty = valid). Checks: id is non-empty and unique, 
    area in allowed set, priority is int 1-9, every trigger metric is in 
    glossary, every op is valid, every value numeric, every depends_on 
    id exists in existing_items.
  - load_backlog(path) / save_backlog(path, data) with indent=2 so git 
    diffs stay clean.

Subcommand: status
  python backlog.py status [--file backlog.json]
  Loads, validates the whole file, then prints a summary: counts per 
  status, then ready/locked/blocked sections each listing id, priority, 
  and trigger in plain text. Sorted by priority then id. Windows-safe, 
  plain ASCII only, no unicode box characters. Exits 0.

Subcommand: add
  python backlog.py add --id ID --title T --area A --priority N 
    --action TEXT [--trigger "manual"] [--depends-on a,b] [--notes TEXT] 
    [--file backlog.json] [--dry-run]
  Builds the item, computes status via determine_status, runs 
  validate_item against current file contents. If any errors: print them, 
  write NOTHING, exit non-zero (atomic). If valid and not dry-run: append 
  to items, set top-level "updated" to today (local date), save, then 
  print the new item's id, computed status, and trigger. --dry-run does 
  all validation and prints the would-be result but does not write.

Default subcommand when none given: status (so bare python backlog.py 
still prints the summary).

PART C — TESTS (pytest, offline, operate on a temp copy of backlog.json 
so the real file is never mutated)

Assert:
  - backlog.json parses, exactly 21 items, all ids unique
  - every item passes validate_item against the rest
  - every depends_on references a real id
  - every trigger metric is in metrics_glossary
  - item 21 has 2 trigger conditions and 2 depends_on entries
  - parse_trigger("manual") == {"all": []}
  - parse_trigger("resolved_count>=30,resolved_count_per_category_max>=15") 
    yields 2 conditions with correct metric/op/value types
  - determine_status: empty trigger + no deps -> ready; data trigger + no 
    deps -> locked; any deps -> blocked
  - add a valid manual item to a temp file -> count +1, status ready
  - add an item with a data trigger -> status locked
  - add an item with depends_on to a real id -> status blocked
  - add with duplicate id -> non-zero exit, file unchanged (count same)
  - add with bad area -> rejected, file unchanged
  - add with trigger metric not in glossary -> rejected, file unchanged
  - add with depends_on to missing id -> rejected, file unchanged
  - python backlog.py status exits 0
pytest exits 0 with output shown.
