## (1) PIPELINE MAP

`main.py` runs an 8-step sequential pipeline. Every step is wrapped in a `try/except` that prints `FAILED` and continues — no step short-circuits the run.

| Step | Label | Module / Function | Reads |
|------|-------|-------------------|-------|
| 1 | Authenticate | `kalshi.authenticate(config)` | `KALSHI_KEY_ID`, `KALSHI_PRIVATE_KEY`, `config.environment` |
| — | Outcome resolution | `logger.resolve_outcomes(config)` | `leviathan.db signals` where `outcome=''`; calls `kalshi.fetch_market()` per unresolved ticker |
| 2 | Fetch markets | `kalshi.fetch_events(config)` → `kalshi.fetch_event_markets(config, ticker)` per event | `config.markets.max_events`, `config.markets.categories` |
| 3 | Filter + score | `scanner.filter_markets()` → `scanner.score_markets()` | `config.markets.*`, `config.whales.*` |
| 4 | Cross-reference | `polymarket.enrich_flagged()`, `external_markets.cross_reference()`, `accounts.enrich_with_smart_money()` | Polymarket Gamma API, Manifold, PredictIt, Metaculus, OddsAPI, Polymarket data-api |
| 5 | Whale + OB | `kalshi.fetch_trades()`, `kalshi.fetch_orderbook()`, `kalshi.fetch_market_history()`, `whales.scan_all_markets()`, `scanner.compute_orderbook_signal()` | Per-ticker trade history, order book, price history |
| 6 | Score with Claude | `scorer.score_markets(flagged_markets, config)` | Spawns `claude --print --system-prompt ... --allowedTools WebSearch` via subprocess; prompt built by `scorer.build_prompt()` |
| 7 | Log | `logger.log_signal()`, `logger.log_run()` | Writes to `leviathan.db` |
| 8 | Send report | `report.compile_report()` → `report.send_report()` | `leviathan.db` stats, `subscribers.json`, `GMAIL_APP_PASSWORD` |

**Scoring via Claude CLI — implications:**
- The subprocess strips `ANTHROPIC_API_KEY` from its environment (`scorer.py:186`) so the CLI uses Pro OAuth. No per-token API billing occurs.
- `config.scoring.scorer_model` and `config.scoring.support_model` are read only for the `run_meta` label and a fallback default string — they are **not passed to the CLI subprocess**. The actual model used is whatever the CLI defaults to for the logged-in account.
- The no-paid-calls-in-loops rule is satisfied: there is exactly one subprocess call per run, regardless of market count.
- Cost is always reported as `$0.0000`. This is correct for Pro, but means cost tracking is permanently inert.

## (3) THE FUNNEL

### Static analysis

```
kalshi.fetch_events()          → up to 400 events (max_events)
  └─ skip KXMVE ticker events  → main.py:95 — event_ticker filter
  └─ fetch_event_markets()     → ~2,458 markets observed
       └─ filter_markets()     → ~32 survivors (observed)
            ├─ skip mve_collection_ticker  ← scanner.py:54 (SECOND MVE filter)
            ├─ volume bounds
            ├─ keyword filter
            └─ close-time bounds
       └─ score_market()       → flag=True on nearly all (see below)
            └─ 32 flagged (observed) — flag is essentially always True
  └─ scorer.score_markets()    → top 10 (max_markets_per_run)
       └─ Claude scores 10, returns direction + confidence
  └─ confidence_threshold=MED  → 0 MED/HIGH signals (recent runs)
  └─ second_pass               → 2 LOW signals logged
```

### (a) Duplicated MVE filtering

Two independent MVE filters run in sequence:

1. **`main.py:95`** — skips events where `event_ticker` contains `"KXMVE"`. This is the effective gate; it runs before any market is fetched.
2. **`scanner.py:54`** — checks `m.get("mve_collection_ticker")` on individual market dicts. This field is not present in Kalshi v2 `/markets` responses (it was a v1 field). The check always evaluates to falsy and does nothing. It is dead code.

### (b) `flag` logic — nearly always True

`score_market()` (scanner.py:241) sets `flag=True` when:
- `raw_edge > 0.08` — requires `base_rate is not None`, which only happens if `estimate_base_rate()` matches one of 8 heuristic patterns. For political/event markets (the dominant type), it returns `None`.
- **`base_rate is None and mid_price is not None`** — this fires on every market that has a price but no heuristic match, which is ~90%+ of all markets passing `filter_markets()`.
- `drift_flag` — fires when OB mid deviates >5% from last traded price.

**In practice:** the upstream volume/time filter in `filter_markets()` is the only meaningful gate. Once a market passes that, it almost certainly gets `flag=True` and goes to Claude. The `edge_threshold=0.08` is irrelevant for the vast majority of flagged markets.

### (c) LOW-confidence second pass

When MED/HIGH scoring produces 0 signals (the common case in recent runs), `main.py:314` runs a second pass including all LOW confidence markets with a non-PASS direction. These are:
- Logged to the DB as new signals (inflating call count).
- Labelled `[SECOND PASS — LOW CONVICTION]` in the report.
- Observed to dominate output: 2 of 2 signals in the last run were second-pass LOW.

The effect is that the system almost never produces zero-signal runs, but the meaningful signal rate (MED/HIGH) appears to be near zero in current market conditions. The second pass masks this.

## (4) PAYOFF / RESOLUTION BUG

**Location:** `logger.py` — `resolve_outcomes()`, line computing `pnl_if_traded`:

```python
pnl = round(float(row["edge"] or 0) * (1 if win else -1), 4)
```

**The bug:** This computes `edge × ±1`, which is meaningless as a P&L figure. For a binary Kalshi contract:

- A YES contract bought at $0.04 that wins pays $1.00 — net profit = $0.96 per contract.
- A YES contract bought at $0.04 that loses — net loss = $0.04 per contract.
- The correct formula for a $1 notional contract is: `win_pnl = 1 - market_price`, `loss_pnl = -market_price`.

**Confirmed in live data:** Two resolved signals show:
- `KXALBUMRELEASEDATEBEY`: bought YES at 2.5%, lost → `pnl_if_traded = -0.075` (should be `-0.025`)
- `KXCABLEAVE-26MAY22-26JUN`: bought YES at 4.5%, lost → `pnl_if_traded = -0.105` (should be `-0.045`)

Both are overstating the loss by the edge amount. Win cases would understate the gain.

**Impact:** `get_stats()` returns a `total_hypothetical_pnl` from these corrupted values. Any future profitability gate (e.g. Kelly sizing, minimum ROI threshold) built on this field will use wrong numbers. The track record display in every report is wrong.

**Fix:**
```python
# In logger.resolve_outcomes(), replace the pnl line with:
market_price = float(row.get("market_price") or 0)
if direction == "YES":
    pnl = round((1 - market_price) if win else -market_price, 4)
else:  # NO contract
    pnl = round(market_price if win else -(1 - market_price), 4)
```

## (5) WHALE / ACCOUNTS INTEGRITY

### Current winner selection logic

`_score_wallet()` (accounts.py:71) computes:
- `avg_pct_pnl` across **all** positions (open + closed)
- `total_cash_pnl` across all positions
- `win_rate` only from positions where `redeemable=True` (market resolved)

`_is_winner()` (accounts.py:130) requires: `position_count >= 3`, `avg_pct_pnl >= 10.0`, `total_cash_pnl >= 25.0`.

**The problem — confirmed by live cache:**

`winning_accounts.json` shows 3 cached winners. All have:
- `resolved_count: 0` — zero resolved positions
- `win_rate: null`
- Active markets: World Cup player props (Ramin Rezaeian 1+ goals: +2387%), Bitcoin Up/Down 5-minute contracts (+367%)

The selection ranks on unrealised PnL of open positions. A wallet holding a 5-minute BTC Up/Down bet that happened to go up 10x is indistinguishable from a skilled forecaster. `resolved_count: 0` means the win-rate gate never fires and these wallets have no verified track record.

**What must change:**
1. Filter to wallets with `resolved_count >= N` (suggest N=10) before scoring.
2. Rank on `win_rate` of resolved positions, not `avg_pct_pnl` of all positions.
3. Exclude markets with titles matching crypto tick patterns ("Bitcoin Up or Down", "Up or Down", "5m", "1m") from the position scoring — these are coin-flip markets where any P&L is noise.
4. Raise `min_cash_pnl` substantially (current $25 is trivially achievable on luck).

## (6) SIGNAL MIX

The two halves of the system target incompatible market types:

**Signals logged (DB analysis):**
- 23 of 24 signals have `market_price < 0.05` (i.e., priced at 1–5%)
- Examples: Taiwan Level 4 warning (2.8%), US territorial acquisition (3.6%), cabinet impeachment (1.4%), Beyoncé album release (2.5%)
- These are long-shot political/event markets. An 8pp edge on a 2% contract means Claude estimated 10% — a 5x disagreement with the market on a low-information tail event.

**Smart money wallets cached:**
- World Cup player prop bets (specific athlete scoring goals)
- Bitcoin 5-minute Up/Down contracts
- These are short-cycle, high-variance, luck-dominated contracts.

**The mismatch:** Neither half is in the intended sweet spot (mid-probability political/economic markets where cross-platform price gaps represent genuine information inefficiency). The signal side is all tiny-probability tail events where an 8pp edge is mostly estimation noise, and the smart money side is tracking lucky gamblers on coin-flip sports props.

**Root cause:** `estimate_base_rate()` returns `None` for most markets, causing `flag=True` on everything with a price. The markets that survive to Claude scoring are whatever passed the volume filter, not a curated set of edge candidates.

## (7) DATA LAYER

### SQLite schema (leviathan.db)

**signals table:**
```sql
CREATE TABLE signals (
    call_id         TEXT PRIMARY KEY,  -- 8-char UUID prefix
    timestamp       TEXT,              -- ISO 8601 UTC
    ticker          TEXT,
    title           TEXT,
    market_price    REAL,
    our_estimate    REAL,
    edge            REAL,
    direction       TEXT,              -- YES | NO
    confidence      TEXT,              -- HIGH | MED | LOW
    whale_detected  INTEGER DEFAULT 0,
    whale_direction TEXT,
    outcome         TEXT,              -- YES | NO | '' (empty = unresolved)
    result          TEXT,              -- WIN | LOSS | ''
    pnl_if_traded   REAL,             -- BUGGY (see section 4)
    run_id          TEXT
)
```

**runs table:**
```sql
CREATE TABLE runs (
    run_id             TEXT PRIMARY KEY,
    timestamp          TEXT,
    markets_scanned    INTEGER,
    signals_generated  INTEGER,
    whale_flags        INTEGER,
    model_used         TEXT,
    tokens_used        INTEGER,
    cost_usd           REAL,
    runtime_ms         INTEGER
)
```

Indexes: `idx_signals_ts` on `timestamp`, `idx_signals_ticker` on `ticker`.

### resolve_outcomes()

Queries all signals with empty `outcome`, calls `kalshi.fetch_market(config, ticker)` for each, checks `market.result` for `"yes"` or `"no"`, then UPDATE the row. Runs synchronously at the start of every pipeline after auth (main.py before step 2). No batching — one API call per unresolved signal.

### Schema extensibility

Adding `contract_type` (binary/scalar/categorical) and `segment` (politics/sports/crypto/macro) can be done with `ALTER TABLE`:

```sql
ALTER TABLE signals ADD COLUMN contract_type TEXT;
ALTER TABLE signals ADD COLUMN segment TEXT;
```

SQLite supports `ADD COLUMN` without a full migration as long as the new column has no NOT NULL constraint without a default. No migration script needed — existing rows will have NULL for both fields, which is fine. `logger.py`'s `log_signal()` would need the two new fields added to the INSERT statement. This is clean.

## (9) OPEN QUESTIONS

1. **What model does the CLI actually use?** `config.scoring.scorer_model` is not passed to the subprocess. The model used is the CLI's account default. Is that Sonnet 4.6, Opus, or something else? This affects output quality and the validity of any A/B comparisons.

2. **Are the 1–5% markets the intended target?** The entire logged signal history is tiny-probability tail events. Is the system supposed to be finding these, or is it supposed to be finding mid-probability markets (30–70%) where cross-platform gaps are more meaningful? This determines whether the funnel needs a `min_market_price` filter or a better `estimate_base_rate()`.

3. **What does "edge" mean for a NO bet on a 2% market?** The system flags `direction=NO` on markets priced at 2–4% and calls it an 8pp edge. But a NO contract at 2% costs $0.98 and pays $1.00 if it resolves NO — the real edge is tiny. Should `edge` be calculated as expected value per dollar risked, not raw probability gap?

4. **Is the second-pass fallback intentional?** It was added to prevent empty reports, but it means nearly every run logs signals — mostly LOW confidence tail bets. If the goal is signal quality over volume, the second pass should either be removed or gated on a minimum absolute edge (e.g. >15pp gap, not just any non-PASS).

5. **Smart money scan scope:** `scan_market()` (accounts.py:228) matches on `condition_id` from Polymarket — which only works if the Kalshi market has a Polymarket match. Markets without a Polymarket equivalent are never smart-money scanned. Is that the intended behaviour?

6. **Who are the intended newsletter subscribers?** The subscriber system is built but `subscribers.json` is empty. The current report format assumes domain knowledge (what "8pp edge" means, what Kalshi is). Does the newsletter need a separate format for non-expert subscribers?

7. **Daily run at 7am — time zone?** `schedule_setup.ps1` uses local time. The server is on EST. Kalshi markets close at various UTC times. Is 7am EST the right cadence, or should INTRADAY markets trigger a separate pre-market scan?
